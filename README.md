# 🐝 Swarm Stigmergy SAR

> **Применение роевых алгоритмов в задачах поисково-спасательных операций**
> Выпускная квалификационная работа · 09.03.01 «Информатика и вычислительная техника»
> ROS 2 Humble · PX4 SITL · Gazebo Harmonic

---

## 📖 О проекте

**Swarm Stigmergy SAR** — исследовательский программный комплекс, моделирующий рой квадрокоптеров, выполняющий задачу поисково-спасательного патрулирования без централизованного управления. Координация достигается за счёт **стигмержи** — механизма непрямого взаимодействия через общее феромонное поле, вдохновлённого поведением социальных насекомых.

Проект построен на индустриальных стандартах робототехники и симуляции: ROS 2 Humble, PX4 SITL и Gazebo Harmonic, — что обеспечивает воспроизводимость результатов и прямой путь переноса прикладного слоя на реальные аппараты.

---

## ✨ Ключевые особенности

🧠 **Полная децентрализация** — каждый агент принимает решения локально, без доступа к глобальному состоянию роя
🌐 **Стигмерджийная координация** — два феромонных поля («visited» и «target») заменяют прямую межагентную связь
🚁 **Реалистичная модель полёта** — штатный PX4 с EKF2 и OFFBOARD-режимом, идентичным реальному автопилоту
🎥 **Сенсорная достоверность** — RGB-камеры в Gazebo с HSV-сегментацией целей
📊 **Автоматизированная экспериментальная база** — серии прогонов с фиксированными сидами и сбором метрик
🔧 **Единая точка конфигурации** — все параметры эксперимента в одном YAML
📈 **Масштабируемость** — проверено до 5 дронов, потолок одной машины ≈ 15–20 агентов
🛡️ **Отказоустойчивость** — падение любого агента не влияет на остальной рой

## 🏛️ Архитектура

Программный комплекс организован как **пятиуровневая слоистая архитектура**, в которой каждый уровень имеет чётко определённую зону ответственности:

```
┌──────────────────────────────────────────────────────────────┐
│  🔬 Уровень 4. Метрики и анализ                              │
│  swarm_metrics → CSV/JSON → plot_results.py → фигуры         │
├──────────────────────────────────────────────────────────────┤
│  🐝 Уровень 3. Координация роя (стигмержи)                   │
│  swarm_pheromone   swarm_explorer×N   swarm_perception×N     │
│                    swarm_agent×N      swarm_controller       │
├──────────────────────────────────────────────────────────────┤
│  🔌 Уровень 2. Транспорт ROS 2 ↔ PX4                         │
│  Micro XRCE-DDS Agent ↔ uORB  ·  px4_msgs                    │
├──────────────────────────────────────────────────────────────┤
│  🚁 Уровень 1. Автопилот (SITL)                              │
│  PX4 × N экземпляров: Commander, EKF2, Position/Attitude     │
├──────────────────────────────────────────────────────────────┤
│  🌍 Уровень 0. Физика и сенсоры                              │
│  Gazebo Harmonic: динамика, камеры, столкновения, мир        │
└──────────────────────────────────────────────────────────────┘
```

Подробное описание каждого уровня, компонентная диаграмма, модель процессов и модель данных — в [`docs/ARCHITECTURE.md`](docs/ARCHITECTURE.md).

## 📦 Пакеты ROS 2

Репозиторий представляет собой ROS 2 workspace со следующими пакетами:

| Пакет | Назначение |
|-------|-----------|
| **`swarm_bringup`** | Launch-файлы, конфигурация экспериментов, запуск SITL и роя |
| **`swarm_interfaces`** | Кастомные сообщения и сервисы (`TargetReport`, `PheromoneGrid`, ...) |
| **`swarm_pheromone`** | Сервер феромонных полей «visited» и «target», правила депонирования, испарения и диффузии |
| **`swarm_explorer`** | Поведенческая модель агента: чтение полей, вычисление вектора движения, захват цели |
| **`swarm_agent`** | Абстракция дрона-агента: связь с PX4, управление состоянием, OFFBOARD-готовность |
| **`swarm_controller`** | Низкоуровневая публикация `TrajectorySetpoint` и управление режимами автопилота |
| **`swarm_perception`** | Обработка RGB-камеры: HSV-сегментация, фильтрация по площади, проекция контура в мировые координаты |
| **`swarm_metrics`** | Сбор телеметрии и отчётов, запись CSV/JSON для постобработки |
| **`px4_msgs`** | Типизированные сообщения uORB ↔ DDS (внешний пакет PX4) |

## 🧩 Ядро системы: алгоритм стигмержи

Сердце проекта — **pheromone-based координация**, где рой ведёт себя как единый распределённый организм:

**🗺️ Поле «visited»** растёт под пролетающими дронами и экспоненциально испаряется — это коллективная «память» роя об уже обследованной территории. Отталкивание от этого поля заставляет агентов автоматически распределяться по пространству.

**🎯 Поле «target»** получает импульсы при обнаружении цели и распространяется гауссовым размытием, создавая «градиент запаха». Притяжение к нему собирает ближайших агентов к месту находки без прямой связи между ними.

**⚖️ Сепарация** от локально наблюдаемых соседей предотвращает столкновения и скучивание.

Движение каждого агента — взвешенная сумма трёх компонентов, а захват цели защищён **гистерезисом контраста**, исключающим «дребезг» между близкими максимумами.

## 🛠️ Технологический стек

| Слой | Технология | Роль |
|------|-----------|------|
| Физика | **Gazebo Harmonic** | динамика, сенсоры, мир 80×80 м |
| Автопилот | **PX4 SITL** | Commander, EKF2, PositionControl |
| Транспорт | **Micro XRCE-DDS Agent** | мост uORB ↔ DDS |
| Middleware | **ROS 2 Humble** | pub/sub граф, namespaces |
| Мост сенсоров | **ros_gz_bridge** | Image, Clock, TF |
| Сообщения | `px4_msgs`, `swarm_interfaces` | контракты данных |
| CV | **OpenCV** (HSV) | сегментация целей |
| Анализ | **Python** (pandas, matplotlib) | метрики и графики |
| Сборка | **colcon** + **CMake** | ROS 2 workspace |

Основные языки реализации — **C++** (узлы реального времени: `swarm_explorer`, `swarm_pheromone`, `swarm_controller`, `swarm_perception`) и **Python** (launch-файлы, скрипты экспериментов и анализа).

## 🚀 Быстрый старт

### Сборка workspace

```bash
# Клонирование
git clone https://github.com/mkvlkvv/APPLICATION-OF-SWARM-ALGORITHMS-IN-CONDUCTING-SEARCH-AND-RESCUE-OPERATIONS.git
cd APPLICATION-OF-SWARM-ALGORITHMS-IN-CONDUCTING-SEARCH-AND-RESCUE-OPERATIONS

# Сборка через colcon
colcon build --symlink-install
source install/setup.bash
```

### Запуск эксперимента

```bash
# 1. Запуск симулятора: Gazebo + N экземпляров PX4 + bridge
ros2 launch swarm_bringup sitl.launch.py

# 2. Запуск инфраструктуры роя: pheromone-сервер, perception, metrics
ros2 launch swarm_bringup swarm.launch.py

# 3. Запуск агентов-исследователей
ros2 launch swarm_bringup explorers.launch.py

# 4. Автоматизированная серия прогонов
python3 scripts/run_experiments.py --config config/experiment.yaml
```

Все параметры (число дронов, целей, сид ГПСЧ, веса алгоритма, длительность прогона) задаются централизованно в `config/experiment.yaml` — это гарантирует **побитовую воспроизводимость** результатов экспериментов ВКР.

## 📊 Метрики эксперимента

Комплекс автоматически собирает три ключевые метрики, предусмотренные ТЗ ВКР:

📐 **Покрытие территории** — доля уникальных обследованных ячеек сетки
⏱️ **Время до первой и последней цели** — скорость реакции и полнота охвата
🎯 **Число обнаруженных целей** — итоговая эффективность роя

Данные пишутся в `runs/<name>_timeseries.csv` и `runs/<name>_summary.json` пакетом `swarm_metrics`. Постобработка выполняется изолированно от real-time контура через `plot_results.py`.

## 🛡️ Надёжность

Система спроектирована с учётом специфики SITL-экспериментов и неидеальности транспорта:

**⏳ Таймаут телеметрии** — при пропадании `vehicle_local_position` более чем на 1 с `swarm_agent` переходит в режим «зависнуть», предотвращая неконтролируемое движение.
**✅ Pre-OFFBOARD прогрев** — перед включением OFFBOARD `swarm_controller` шлёт ≥10 setpoint-ов, удовлетворяя требованию PX4.
**🔒 Гистерезис захвата** — исключает конкуренцию двух агентов за один максимум.
**🧯 Изоляция сбоев** — падение одной `swarm_explorer`-ноды не затрагивает остальных благодаря чистой pub/sub-архитектуре.

## 🎓 Академический контекст

Проект выполнен в рамках выпускной квалификационной работы по направлению **09.03.01 «Информатика и вычислительная техника»** и демонстрирует применение принципов **коллективного интеллекта** к актуальной задаче поисково-спасательных операций.

Архитектурные решения следуют современным практикам проектирования робототехнических систем: слоистая декомпозиция, единый транспорт DDS, контракты данных через типизированные сообщения, полная воспроизводимость через централизованную конфигурацию.

Выбор PX4 как модели автопилота обеспечивает **прямой путь переноса** прикладного слоя на реальные аппараты: интерфейсы `TrajectorySetpoint` и `VehicleLocalPosition` идентичны в SITL и на физическом Pixhawk.
